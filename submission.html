01 — Project Description (copy-paste ready)

Short description (140 chars)
Trustless B2B escrow on Solana with AI dispute resolution. Replaces months of arbitration with a verdict in seconds. x402-compatible for AI agents.
Full project description
ProofPay is a B2B escrow protocol built on Solana that replaces human arbitrators with an AI oracle — cutting dispute resolution from months to seconds, and from R$130,000 in fees to the cost of a gas transaction. THE PROBLEM B2B contract disputes are expensive and slow by design. In Brazil alone: - The judiciary takes 2–4 years to resolve a contract dispute - Institutional arbitration costs ~R$130,000 for a R$10M contract - 73% of SMEs are impacted by payment delays tied to disputes - The AAA resolved $21.3 billion in B2B disputes in 2024 — the market is enormous and broken THE SOLUTION ProofPay locks funds in a Solana PDA vault at contract signing. When a milestone is delivered and accepted, the payer releases funds directly to the payee. If delivery is disputed, our AI oracle (powered by Claude) evaluates the evidence and executes the resolution on-chain — in seconds, not months. HOW IT WORKS 1. Payer creates an escrow on-chain (create_escrow), defining payee, oracle, milestones, and timeout 2. Funds are locked in a vault PDA controlled exclusively by the program 3. On delivery: payer calls release_milestone → USDC transferred directly to payee 4. On dispute: either party calls open_dispute → state machine freezes funds 5. Our oracle server receives the dispute evidence, calls the Anthropic API, and signs resolve_dispute on-chain 6. If the oracle fails to respond within 7 days, fallback_dispute_resolution automatically returns funds to the payer WHAT WE BUILT - Anchor smart contract (Rust): 8 instructions, full state machine, checked arithmetic throughout, PDA rent optimization (close = payer on terminal states) - AI Oracle server (Hono + Node.js on Render): calls Claude API, signs on-chain resolution with oracle keypair - React dashboard: create, monitor, and dispute escrows — live on Vercel - TypeScript SDK (@proofpay/sdk): ProofPayClient + x402 utilities - Supabase schema: escrows, milestones, event_logs, oracle_decisions with RLS and triggers - 4 test suites: setup, happy path, security invariants, dispute resolution THE AGENTIC ANGLE ProofPay natively implements the x402 Protocol (HTTP 402 Payment Required). Any AI agent — LangChain, CrewAI, Eliza — can discover a ProofPay-protected service, negotiate payment terms, and settle in USDC autonomously in ~400ms. No human co-signer. No bank account. Just Solana. This makes ProofPay the settlement infrastructure layer for the AI agent economy. SECURITY DESIGN Every instruction follows CHECK → EFFECT → INTERACT. State transitions are enforced by require!() guards. All arithmetic uses checked_mul/checked_sub (no overflow possible). The dispute state machine ensures funds can never be moved while a dispute is open — release_milestone reverts with InvalidState if called during dispute. Live on Solana devnet. Open source. MIT license.

04 — Judge Q&A — Perguntas Difíceis + Respostas Fortes

Q: "Why does this need to be on-chain? Can't you just use Stripe + a contract?"
Stripe holds the money and can reverse it. A lawyer enforces the contract. ProofPay eliminates both intermediaries: the smart contract is the escrow (non-custodial, non-reversible by anyone except the program logic), and Claude is the arbitrator. For AI agents, there's no identity, no bank account, no legal entity to sign a contract with. On-chain is the only path.
Q: "What happens if Claude gives a wrong verdict?"
The oracle evaluates structured evidence provided by the parties. The decision is final and on-chain — same as a binding arbitration clause. The design is intentional: by making decisions final, we remove the attack surface of "appeal indefinitely until the other party gives up." If the oracle times out (doesn't respond in 7 days), fallback_dispute_resolution automatically returns funds to the payer — there's always a deterministic outcome. For high-value disputes, the next version will support human-in-the-loop review via oracle_decisions.status = 'pending_human_review' (already in the schema).
Q: "The oracle is centralized — it's a single keypair on Render. That's a point of failure."
Correct, and we document this explicitly in PROOFPAY_POST_HACKATHON.md as a mainnet blocker. For the hackathon, the oracle keypair is the trusted arbitrator by design (same as you trust AAA as the arbitration body). The decentralization path is a multisig oracle network where 2-of-3 AI evaluations must agree before signing resolve_dispute. The Anchor program already supports any oracle Pubkey — swapping the keypair for a multisig is a server-side change, not a contract change.
Q: "Does this actually work end-to-end? Show me."
Yes. The oracle server is live at https://proofpay-oracle.onrender.com (health: /health). The program is deployed at FpN5kH3w6kVLDEHz1zUfSof2n2QfMKfENCE97LMiut6i on devnet. The frontend is on Vercel. We have 4 test suites (anchor test) covering: setup, happy path create→fund→release, security invariants (Disputed blocks release_milestone with error 6000), and full dispute resolution with oracle keypair. Every transaction is verifiable on Solscan.
Q: "Who are your competitors in the Solana ecosystem?"
Generic escrow programs exist, but none combine: (1) AI dispute resolution that executes on-chain, (2) x402 Protocol for AI agent payments, (3) focus on B2B contract enforcement specifically. The closest analog is traditional arbitration providers like AAA or Kleros (Ethereum), not Solana-native. Our differentiation is the AI oracle that makes resolution as fast as an API call, not as slow as a legal process.
Q: "What's the x402 angle? Is that actually implemented or just mentioned?"
The x402 Protocol is implemented in sdk/src/index.ts: parseX402Challenge() parses HTTP 402 response headers, buildX402PaymentHeader() constructs the X-PAYMENT header from a signed Solana transaction, and the server's dispute-oracle Edge Function proxies x402 challenges. The SDK exports X402Challenge and X402PaymentPayload types. What's NOT yet done: the frontend doesn't have an x402 flow — that's Phase 2, where AI agents hit a ProofPay-protected API endpoint and auto-negotiate payment. The primitives are ready; the UI integration is the next step.
Q: "Why Solana? This could run on any EVM chain."
Three reasons specific to our use case: (1) ~400ms finality means the oracle can resolve a dispute and the user sees confirmation before they close the browser tab — on Ethereum that's 12+ seconds; (2) gas costs make micro-contracts viable — a $100 freelance job with a $0.001 gas fee makes sense, not so much with $5–$20 EVM gas; (3) the x402 ecosystem is being built natively on Solana — the Coinbase x402 library targets Solana first. We're building in the right ecosystem for the AI agent economy.
Q: "PMEs in Brazil — will they actually use a Solana wallet?"
Today, no. The current version targets two audiences: (1) the hackathon/grant evaluators who understand Solana, and (2) AI agents that don't need a UI at all — they interact via SDK. The PME B2B market is the long-term opportunity, and it requires abstraction layers (embedded wallets, USDC on/off-ramps, BRL integration). We've documented this in PROOFPAY_BUGS.md as a known adoption barrier. The protocol is sound; the UX layer is the roadmap.

State Machine Diagram

Created ──[fund_escrow]──► Funded ──[release_milestone × N]──► Completed (PDA closed)
                              │
                              ├──[refund_on_timeout]──────────► Refunded  (PDA closed)
                              │
                              └──[open_dispute]────────────────► Disputed
                                                                     │
                                                   [resolve_dispute] ├──► Completed (release_to_payee=true)
                                                                     │
                                                   [resolve_dispute] ├──► Refunded  (release_to_payee=false)
                                                                     │
                                              [fallback_dispute_res] └──► Refunded  (after 7 days)

Key Anchor Patterns

  • seeds = [b"escrow", escrow_id] — deterministic PDA per contract, no collision possible
  • token::authority = escrow — vault controlled exclusively by program, no admin key
  • close = payer on terminal states — eliminates orphan PDA accumulation
  • has_one = usdc_mint on all transfer instructions — no token substitution attacks
  • Release amount uses BPS math: (total * bps) / 10000 with 128-bit intermediate — no rounding loss

EscrowAccount Memory Layout

MAX_SIZE = 8 (discriminator) + 597 (fixed fields) + 664 (10 milestones max) = 1269 bytes

State byte at offset 255 in raw account data: 0=Created, 1=Funded, 2=Completed, 3=Refunded, 4=Disputed
